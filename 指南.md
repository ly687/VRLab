# VRLab GitHub Pages 静态托管指南

## 先给结论

可以做到。

VRLab 是 Vite + TypeScript + Three.js 的纯前端项目，没有后端服务，构建后会生成静态文件，因此适合部署到 GitHub Pages。

免费情况：

- 如果仓库是公开仓库，GitHub Free 账号可以免费使用 GitHub Pages。
- 如果仓库是私有仓库，GitHub Pages 通常需要 GitHub Pro、Team、Enterprise 等付费方案支持。
- GitHub Pages 站点发布后会公开访问，即使仓库本身是私有仓库，也要避免把敏感内容放进发布产物。

本项目仓库是：

```txt
https://github.com/ly687/VRLab
```

部署成功后的默认访问地址通常是：

```txt
https://ly687.github.io/VRLab/
```

官方参考：

- GitHub Pages 说明：https://docs.github.com/en/pages/getting-started-with-github-pages/what-is-github-pages
- GitHub Pages 发布源配置：https://docs.github.com/en/pages/getting-started-with-github-pages/configuring-a-publishing-source-for-your-github-pages-site
- Vite 静态部署说明：https://vite.dev/guide/static-deploy.html#github-pages

## 为什么 VRLab 适合 GitHub Pages

VRLab 的运行方式是：

1. 源码在 `src/`。
2. 执行 `npm run build`。
3. Vite 生成 `dist/` 静态文件。
4. 浏览器访问静态 HTML、CSS、JS。

GitHub Pages 正好用于托管 HTML、CSS、JavaScript 这类静态文件。

项目中的这些内容不需要上传到 GitHub Pages：

- `node_modules/`：依赖安装时自动生成。
- `dist/`：构建时自动生成。
- `docs/`：当前要求不推送。
- `参考资料/`：当前要求不推送。

## 推荐部署方案

推荐使用 GitHub Actions 自动构建并发布。

原因：

- VRLab 是 Vite 项目，必须先运行构建命令生成 `dist/`。
- 不需要把 `dist/` 提交进仓库。
- 每次推送到 `main` 后可以自动部署。
- 不需要使用 `docs/` 目录，符合当前仓库规则。

## 第一步：添加 Vite 配置

当前项目没有 `vite.config.ts`。由于 GitHub Pages 项目站点的路径是 `/VRLab/`，需要显式配置 Vite 的 `base`。

在项目根目录新建：

```txt
vite.config.ts
```

内容如下：

```ts
import { defineConfig } from 'vite';

export default defineConfig({
  base: '/VRLab/',
});
```

为什么需要这一步：

- 本地开发地址通常是 `http://localhost:5173/`。
- GitHub Pages 地址是 `https://ly687.github.io/VRLab/`。
- 如果不设置 `base: '/VRLab/'`，构建后的 JS/CSS 资源可能会从 `https://ly687.github.io/assets/...` 加载，导致页面空白或 404。

## 第二步：添加 GitHub Actions 部署文件

在项目根目录新建目录和文件：

```txt
.github/workflows/deploy.yml
```

内容如下：

```yaml
name: Deploy VRLab to GitHub Pages

on:
  push:
    branches:
      - main
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - name: Install dependencies
        run: npm ci

      - name: Build
        run: npm run build

      - name: Setup Pages
        uses: actions/configure-pages@v5

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./dist

      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

这个工作流会在每次推送 `main` 分支时执行：

1. 拉取仓库代码。
2. 安装 Node.js。
3. 用 `npm ci` 安装依赖。
4. 运行 `npm run build`。
5. 上传 `dist/`。
6. 发布到 GitHub Pages。

## 第三步：本地先验证构建

提交前先在本地运行：

```bash
npm install
npm run build
```

如果要预览构建后的效果：

```bash
npm run preview
```

注意：`npm run preview` 默认仍然是在本地预览，实际线上路径以 GitHub Pages 地址为准。

## 第四步：提交并推送配置

确认 `.gitignore` 仍然排除这些目录：

```txt
node_modules/
dist/
docs/
参考资料/
```

然后提交：

```bash
git add vite.config.ts .github/workflows/deploy.yml 指南.md
git commit -m "Add GitHub Pages deployment guide"
git push origin main
```

推送完成后，GitHub Actions 会自动开始构建和部署。

## 第五步：在 GitHub 开启 Pages

进入 GitHub 仓库页面：

```txt
https://github.com/ly687/VRLab
```

然后操作：

1. 打开 `Settings`。
2. 左侧找到 `Pages`。
3. 在 `Build and deployment` 里找到 `Source`。
4. 选择 `GitHub Actions`。
5. 保存设置。

之后查看 `Actions` 页面，等待部署任务完成。

部署成功后，页面地址一般是：

```txt
https://ly687.github.io/VRLab/
```

## 摄像头相关注意事项

VRLab 需要浏览器摄像头权限。

GitHub Pages 默认使用 HTTPS，因此可以正常请求摄像头权限。浏览器通常只允许以下来源调用摄像头：

- `localhost`
- `https://` 页面

所以部署到 GitHub Pages 后，摄像头权限是符合浏览器安全要求的。

使用者打开线上页面后，需要点击开始按钮并允许摄像头权限。

## MediaPipe 资源注意事项

项目使用 MediaPipe Tasks Vision 进行手部追踪。首次加载时可能需要下载 wasm 和模型资源。

如果用户网络无法访问相关资源，页面可能能打开，但手部追踪初始化会失败。

排查方式：

- 换 Chrome 或 Edge。
- 检查浏览器控制台报错。
- 确认网络能访问 MediaPipe 资源。
- 确认摄像头权限已允许。

## 常见问题

### 页面打开后一片空白

优先检查是否已经配置：

```ts
base: '/VRLab/'
```

如果没有配置，GitHub Pages 子路径下的资源路径很容易错误。

### Actions 构建失败

进入 GitHub 仓库的 `Actions` 页面，打开失败的 workflow，重点看：

- `npm ci` 是否失败。
- `npm run build` 是否失败。
- 是否有 TypeScript 编译错误。

本地也可以先运行：

```bash
npm run build
```

### 摄像头权限没有弹出

检查：

- 是否使用 HTTPS 地址。
- 浏览器是否已经禁用了摄像头权限。
- 是否有其他软件占用摄像头。
- 是否使用 Chrome 或 Edge。

### 可以把 dist 直接推到 GitHub 吗？

可以，但不推荐。

当前推荐方式是用 GitHub Actions 构建并发布 `dist/`，这样仓库只保留源码和配置，构建产物不进入主分支。

### 可以使用 docs 目录发布吗？

不推荐。

原因：

- 当前项目规则要求 `docs/` 不推送。
- Vite 项目需要构建产物，直接用 `docs/` 不如 GitHub Actions 清晰。
